Creating A New Experiment

Overview

The following picture presents a summary of the information flow associated to the ReC/e-lab project, as well as all participants in that flow. In the left side, there will be the experiment's GUI. In the middle, there is the multicast, which is the core of the system, serving as a mediator for the connections between clients and the hardware/apparatus. On the right-most side, we have the hardware itself and, on its left, there will be the drivers that allow the communication with the hardware.

New experiment

Step 1 - Creating the structure for a new experiment

Create a new project for the experiment, using the following command:

ant -f build_new_experiment.xml create.experiment -D experiment.name=<experiment_name>

in $REC_HOME/templates/ (there will be an ant file, named build_new_experiment.xml in that directory).The name of the experiment should be small, such as an acronym, without any spaces or special characters such as 'ç', 'ã', etc.The previous statement will create a new folder in $REC_HOME/experiments/, with the as the base folder. Its structure will be the following (all directories will be empty, and there will be a small readme file with a short description with the contents of each folder):

  • docs - directory containing general documentation related with the experiment, such as protocols (originals from producers), apparatus descriptions, specific datasheets, etc.
    • CommonDatasheets – datasheets related to components and devices that could be used in more than one experiment. If the datasheet is related only to a particular experiment, it should be moved to Apparatus\Docs.
    • e-escola - original doc/html or txt documents from e-escola topics
      • examples - reports and data analysis of each experiments.It can be .xls, .doc, .pdf, etc.
      • presentation - presentations related to the experiment's subject; it should contain original material to be treated by teachers as supporting material for lessons.
    • microcontroller - documentation and zip source files for the microcontroller experiment.
      • codesources - repository of .zip files containing all source and libs necessary for compiling and running the code on the microcontroller experiment.
      • PCB - specific designs of printed circuit boards and plugs/connectors for each experiment.
    • pictures - pictures and schematics sources (like .fla or Java scips) from each experiment.
  • etc - it should contain one build.xml to override specific experiment build configurations, as well as its corresponding build.properties file. It also contains the ReCBaseUIConfig.xml and ReCFaceConfig.xml GUI configuration files, whose contents will be explained later.
  • lib
    • java - in this directory, you should place the jar files that should be available for both server and client parts of this specific experiment, not already included in the base libraries for ReC or e-lab basic structures.
      • client - in this directory, you should place the jar files that should be available ONLY for client parts of this specific experiment, not already included in the base libraries for ReC or e-lab basic structures.
      • server - in this directory, you should place the jar files that should be available ONLY for server parts of this specific experiment, not already included in the base libraries for ReC or e-lab basic structures.
  • src
    • java
      • client - the customizer's code for this experiment. You may extend the platform base code both from e-lab and ReC as needed, but the sources should be located here for build system correct integration.
      • server - the driver's code for this experiment. You may extend the platform base code both from e-lab and ReC as needed, but the sources should be located here for build system correct integration.
    • pic - in this directory, you should place the source files for the PIC code for this experiment controller.
    • stamp - in this directory you should put the source files for the basic stamp 2 code for this experiment controller.

Notes:

  • The previous structure should not be changed or the deployment system will stop working
  • If there is already an experiment with the same name, nothing will be created/deleted/overwritten.

Additionally, the previous ant target updates the file $REC_HOME/templates/build.experiments.properties, adding the created experiment to its list. This file should not be changed manually.

Step 2 - Configuring the server/driver

Create the server side classes

Create the *ServerMain* class in a package such as pt.utl.ist.elab.driver. which should contain something similar to:

public class ServerMain {

private static String <EXPERIMENT>_HARDWARE_LOGGER = "<Experiment>Hardware.Logger";

static {
final Logger l = LogManager.getLogManager().getLogger(ServerMain.<EXPERIMENT>_HARDWARE_LOGGER);
if (l == null) {
LogManager.getLogManager().addLogger(Logger.getLogger(ServerMain.<EXPERIMENT>_HARDWARE_LOGGER));
}
}

/**
* @param args the command line arguments
*/
public static void main(final String[] args) {
try {
ORBBean.getORBBean();

new BaseHardware(new <Experiment>Driver());

try {
Thread.currentThread().join();
} catch (final Exception ignored) {
}

ORBBean.getORBBean().killORB();
} catch (final Exception e) {
LoggerUtil.logThrowable("Error on Main...", e, Logger.getLogger(ServerMain.OPTICA_HARDWARE_LOGGER));
}
}
}

and the corresponding driver class:

import pt.utl.ist.elab.driver.serial.serialportgeneric.genericexperiment.GenericSerialPortDriver;

public class OpticaDriver extends GenericSerialPortDriver {
}

Displays

Customizer

The customizer, placed in $experiment_name/src/java/client/. E.g.,

$REC_HOME/experiments/myexperiment/src/java/client/pt/utl/ist/elab/client/myexperiment

is the Java GUI containing the customization options for the experiment. This is one of the pieces that needs to be created from scratch, for each experiment, with their own specific characteristics and customization parameters.

  • Graphics
  • Tables
  • Any specific (not reusable) panel

Step 4 - Building/packaging the experiment

The build system, like the project, is organized in the following major groups:

  • ReC Common
  • ReC Client (shared client and UI)
  • ReC Hardware Server
  • ReC Multicast Controller
  • Elab Client
  • Elab Driver
  • Elab Multicast Controller

There are targets for each of the previous groups, but basically do the same tasks:

  • clean - This target will delete the generated compiled classes from the corresponding group part of the project. The classes will be located in respective folder under /build structure. Corresponding source code and generated jar (signed or not) will not be deleted.
  • compile – compiles the project to the build folder (to the corresponding sub-folder as target) using as classpath the necessary dependencies (each part may only depend on the common part of the project), as well as the libraries specified in the correct lib structure.
  • jar – it packages each part of the project in its own jar file, with the corresponding manifest file specifying the Main class, when appropriately.
  • sign – it uses the keystore properties defined in build.properties to sign the generated jars, the only jars that need to be signed are common and client side part of the projects, therefore, this are the only jars that are signed. It generates a file JAR_NAME_BEFORE_SIGNED_sig.jar.
  • deploy
  • dist – it performs a clean compilation, packaging, and signs the jar files that need to be signed (client and shared). Additionally, there are also some targets to manage library dependencies and JNLP generation:
  • Libraries
    • sign.libraries
    • deploy.libraries
  • JNLP
    • create.jnlp.for.experiment
    • create.jnlp.for.experiments.in.list

There are specific targets for clean, compile, jar and sign, for each experiment, just like in the main project (ReC). To do so, all you have to do is specify the name of the experiment (-Dexperiment.name=). To use the process list targets, specify the property experiment.list in the build.properties file.

  • Experiments
    • clean.experiments
    • clean.experiments.list
    • compile.experiments.list
    • compile.experiment
    • jar.experiments.list
    • jar.experiment
    • sign.experiments.list
    • sign.experiment

Additionally there are general targets available. Those are no more than a sum of their representation in the group targets. Therefore, there are a clean all target, a compile all target, and so forth.To build your new experiment, you should first configure the necessary properties

# One experiment must always be associated to one and just one laboratory labx.experiment.list=myexperiment # Connection to the laboratory (Multicast) labx.multicast.controller.host=elab.ist.utl.pt labx.multicast.controller.port=9001 # Identification of the laboratory in the Allocation Manager, used to check access to an experiment labx.rec.am.lab.id=LABX

and execute the following statements, next:

ant -f buildall.xml dist.all -Dlab.list=labx cd $REC_HOME/rec.am mvn clean package

The previous statements will result in something like:

$REC_HOME/dist/java/myexperimentHarwareServer_10-03-2011.zip $REC_HOME/dist/java/multicast_labx_10-03-2011.zip $REC_HOME/rec.am/target/rec.am.war

Step 5 - Deploying

DriverHardware unzip

The experiment's zip file from the previous step (myexperimentHarwareServer_10-03-2011.zip), should be copied to the deployment server, and unzipped into something like /home/elab/rec-deployment/myexperiment.Each experiment is self contained, meaning it needs to have its own directory. Notice that, after the unzipping operation, the unzipped structure should not be changed.

Enabling the experiment in the GUI

To make an experiment "visible" to the GUI, you must update the JNLP file in the Application server, in this case, Glassfish. To do so, you need to go to the server where Glassfish is installed and, probably, running, and execute the following statement (make sure you copied previously the new version of the allocation manager WAR file, generated in step 4, into the server):

cp $REC_HOME/rec.am/target/rec.am.war glassfishv3/glassfish/domains/domain1/autodeploy/

and make sure glassfish the deployment in glassfish was successful

Driver startup

For each experiment, there is one script with the following options:

  • start - it starts an experiment
  • stop - it stops an experiment
  • restart - it stops and then starts an experiment
  • status - it checks the status of an experiment (one of running or stopped)

The script will be named as myexpermentDeamon.sh and it gets the operation name as parameter. E.g.,

/home/elab/rec-deployment/myexperimentDeamon.sh status
  • Note: Remember to be in the appropriate node in the server to start the experiment! Check the configurations in the Cluster Setup page

Driver logging

The driver records logging messages in files to make it easier to understand the driver's status and what it is doing. These messages will be availble in the files named myexperiment_*.log.The logging tool in the project is the well known Log4J and one of its characteristics is the possibility to configure the logging level. All these configurations are performed in the file hardwareserver/etc/loggers.config.properties of the unzipped folder
